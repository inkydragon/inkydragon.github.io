基于 [[../drafts/05-26-ai-era-thoughts.md|AI时代开源思考]]、[[../drafts/05-26-ai-era-open-source.md|AI时代开源方向]]、[[个人项目选择规则.md|个人项目选择规则]] 综合提炼。

## 视角一：个人发展（ChemE PhD / PSE 过程系统工程研究者）

### 准入判据（两条铁律）

1. **AI 做不好，且短期内做不好** → 才有进入价值
2. **产出 3 年后仍有价值** → 才值得投入时间

两个都"是"的方向值得深入；只中一个，需谨慎。

### 优先级排序

| 优先级 | 条件 | 代表方向 |
|--------|------|----------|
| **P0** | 毕业/研究/简历相关 + 2-6周可展示 + AI可放大 | AI + 化工PSE（软测量、异常诊断、知识变量建模、实验自动化） |
| **P1** | 自己会用 + 改善日常工作流 + 可逐步演化 + 不需外部反馈 | 实验harness、文献整理、任务执行辅助、数值工具 |
| **P2** | 技术有趣 + 明确短期产出 + 不承诺长期维护 + 做完可归档 | GEMM kernel、数学函数实验、AI能力探测demo |
| **P3** | 自己不用 + 无明确用户/产出 + 仅"方向有意义" | 不做 |

### 启动检查表（五问）

1. **需求** — 我的真实需求？未来一个月会用它吗？
2. **产出** — 文章/demo/benchmark/论文实验/简历项目？别人能看懂？
3. **终点** — 什么叫完成？最小版本？何时可归档？
4. **god-food** — 我能否持续受益？社区无反响我还继续吗？
5. **AI放大** — AI具体加速什么（样板代码/实验/测试/文档/调试）？有无自动验证防垃圾？

五问回答不清楚 → 不开项目。

### 个人独特优势（护城河）

- **领域知识壁垒**：化工系统工程、过程监测、工业标准——AI训练数据中稀疏、难获取的结构化知识
- **跨领域组合**：ChemE 领域知识 + AI工具链 + 数据建模 + 工程实现 = 罕见交集
- **验证闭环**：实验室/工业数据提供真实反馈，不是纯代码游戏

## 视角二：开源社区贡献 — 项目投入筛选判据

> **核心判断：代码不再是稀缺资源，判断力才是。**
>
> 贡献的价值重心从"生产代码"转向"生产判断和验证正确性"。
> 审查成本没有随生成成本一起坍塌——这是 AI 时代人类贡献的支点。

---

### 第一层：价值判据 — 这个项目/方向值不值得看？

**判据 1：产出属性**
> 核心产出是"判断"还是"代码"？

| 判断型（增值） | 代码型（贬值） |
|---------------|---------------|
| 设计一个测试集，定义"好"的标准 | 写一个功能实现 |
| 梳理领域常识，结构化知识库 | 封装一个库 |
| 审计安全面，分类攻击模式 | 修一个常规 bug |
| 定义协议/接口/兼容性契约 | 写胶水代码 |

- 如果 AI 按需就能生成等价产出 → **不进**
- 如果产出能被他人（或 AI）引用为决策依据 → **值得**

**判据 2：AI 替代期限**
> AI 在多长时间内能做得好这个项目的核心工作？

| AI 能做好的时间 | 判断 |
|----------------|------|
| ≤ 6 个月 | 不值得投入（除非有极强先发优势） |
| 3–5 年不确定 | 人类判断力窗口期，值得 |
| ≥ 5 年或根本不可能 | 高价值方向（涉及物理约束、法规、共识、品味） |

**判据 3：裁员测试**
> 如果代码明天被 AI 完全重写，剩下的东西还有没有价值？

- **有** → 剩下的（规格、测试集、知识库、社区共识）是真正护城河 → **进**
- **没有** → 项目本质是代码的实现细节 → **不进**

---

### 第二层：需求判据 — 这个方向在 AI 时代是增强还是减弱？

| 需求趋势 | 信号举例 | 判断 |
|----------|---------|------|
| **增强** | 形式化验证需求随AI代码量增长而爆发 | **优先** |
| **增强** | 安全审计面随AI依赖链加深而扩大 | **优先** |
| **减弱** | 便携数学库可被AI按需生成替代 | **不进** |
| **减弱** | 通用CRUD库、中间件封装 | **不进** |
| **翻转** | 依赖从"省时间"变成"增负债" | **警惕依赖重的项目** |

---

### 第三层：资产判据 — 项目有什么不可替代的积累？

| 资产类型 | AI 能替代吗 | 是否有网络效应 | 判据 |
|----------|------------|---------------|------|
| 代码实现 | **能** | 弱 | 不可靠资产 |
| 规格/标准 | **不能** | 强（别人依赖你的定义） | **核心护城河** |
| 测试集/基准 | **不能** | 强（评测依赖测试集） | **核心护城河** |
| 领域知识库 | **不能** | 中 | **护城河** |
| 边界case数据 | **不能** | 中（bug是独特数据） | **护城河** |
| 社区共识/品牌 | **不能** | 强 | **护城河** |
| 文档/tutorial | **部分能** | 弱 | 脆弱资产 |

> 问：项目积累的资产中，不可替代型占多大比例？
> 比例越低 → 越不值得投入。

---

### 第四层：参与判据 — 这个项目值得投入时间吗？

**社区健康度检查：**

- [ ] Star 在 1万–10万 之间（太小缺乏生态影响力，太大贡献门槛高）
- [ ] 社区活跃（issue 响应时间 ≤ 72小时）
- [ ] 有清晰的贡献指南
- [ ] 有明确的 AI 贡献政策（欢迎、要求披露、还是禁止）
- [ ] 维护者还在积极回应

**个人匹配度检查：**

- [ ] 有独特的领域知识可以贡献（不是谁都能写的那种代码）
- [ ] 能增进对某个领域的深入理解（不只是刷PR数）
- [ ] 即使项目没达到预期，过程有副产品价值（测试数据、知识笔记）

### 第五层：退出判据 — 什么时候该放弃/降级一个项目？

**放弃信号（果断退出）：**
- 核心价值可被 AI 按需生成替代
- 社区已不活跃，没有人在依赖它
- 维护成本持续高于收益
- 自己不再使用，也没有明确用户群

**保留信号（值得继续）：**
- 积累了边界 case 数据集（bug 是独特资产）
- 有活跃社区共识，文档/规格是其他人的依赖
- 决策过程本身就是公共品（治理记录、RFC讨论）

**最难区间（需主动决策）：**
> 中间层的库——不是特别有名、没有独特数据壁垒、但还在被使用。
> → 需要人主动决定：体面归档，还是用 AI 压缩维护成本继续？

**降级策略：**
1. 进入 maintenance mode：不主动扩展、不承诺 roadmap、只修 bug、只合高质量 PR
2. 在 README 写清楚当前状态
3. 如果长期无反馈 → 归档

---

### 高价值领域速查（天然抗 AI）

| 领域 | 为什么 AI 做不好 | 核心产出类型 |
|------|------------------|-------------|
| **基准与评测** | 定义"好"的标准本身是判断，不是代码 | 规格 |
| **验证工具链** | 验证方法的设计需要理解什么会出错 | 工具+方法 |
| **领域知识结构化** | 隐式知识不在训练数据中 | 知识库 |
| **标准与接口** | 设计量 >> 代码量，需要共识 | 规格 |
| **共识与治理工具** | 涉及人的决策和信任 | 流程+工具 |
| **安全与供应链审计** | 威胁建模需要创造性思维 | 审计方法论 |
| **"最后一公里"经验** | 每个场景的集成模式是独特的 | 经验模式 |

### 两级过滤器（极简版）

当时间有限、需要快速判断时：

**第一级：方向级**
1. **AI 做不好或做不了，且短期内做不好？** → 是，才有进入价值
2. **产出的东西去掉代码后，3 年后还有用？** → 是，才值得投入时间

**第二级：项目级**
1. **项目积累的资产是"判断"还是"代码"？** → 判断型优先
2. **社区还活着吗？** → issue 响应 ≤ 72h，维护者活跃
3. **我有别人没有的东西吗？** → 独特领域知识、边界case理解、审美/品味

两级都通过 → 值得投入。
只通过一级 → 谨慎。
都不通过 → 不进。

---

### 贡献起点策略

进入一个新项目时的优先级（从低到高）：

1. **审文档** → 建立系统理解的入口
2. **分类 issue** → 理解项目的痛点和边界 case
3. **写测试** → 测试即规格；AI 做得最差，项目最需要
4. **审 PR** → 审查能力是新超能力；项目更需要审查者而非贡献者
5. **设计/治理** → 凝聚共识、定义标准、安全审计——人类不可替代

> 从小事做起，不是从代码写起。代码是最后一步。

## 两视角的交集：什么方向同时满足个人发展和社区贡献？

| 交集方向 | 个人价值 | 社区价值 |
|----------|----------|----------|
| **化工PSE + 基准/评测** | 简历+论文+领域知识壁垒 | 评测基础设施薄弱，工业AI评测稀缺 |
| **领域知识结构化** | 体系化自己的领域知识 | 将隐式工程知识变成可查询的公共品 |
| **验证工具链（面向科学计算）** | 数值计算正确性验证是PSE核心痛点 | 科学计算验证工具普遍稀缺 |
| **AI治理/贡献政策** | 跨领域叙事：ChemE + AI时代的工程伦理 | 社区急需，AI完全做不了 |
| **"最后一公里"经验模式化** | 将PSE工作流中的隐性知识显式化 | 决定AI能否在项目中真正用起来 |
