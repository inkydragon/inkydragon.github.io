## 逐项分析

### 1. AI驱动形式化验证（core-math）

**两级过滤器：**
- 第一级 ✅✅：AI 做不好（形式化证明是 AI 最短板），去掉代码剩下证明/规格 → 3年后仍有价值
- 第二级 ✅✅⚠️：判断型资产（证明=判断）、社区活跃（core-math 是活跃项目）、个人匹配度存疑

**手艺性评估（关键缺陷）：**
你的项目诊断已经指出了核心问题：证明搜索的"随机阻塞"特性天然反心流。一个 lemma 卡几小时是常态——这是解谜（puzzle），不是手艺（craft）。

> 前 AI 时代"写代码"这层手艺活填充了证明搜索之间的等待。现在这层被 AI 剥离了，剩下纯证明太硬、太脆，不适合放松。

**作为"缓解 AI 焦虑"的项目：**
这里有个微妙的问题。形式化验证在战略上最抗 AI——但它恰好把 AI 最不擅长的工作（证明搜索）留给了你。你**用 AI 辅助证明**的想法是对的，但当前 AI 辅助证明搜索（Coq/Lean 的 AI tactic）还在早期，不太靠谱。

**判定：** 战略价值最高，但作为闲暇放松项目有结构性缺陷。更适合作为 **Sprint 探索**——选一个函数，用 AI 辅助验证，写一篇文章，探路后决定。不是长期手艺项目的好选择。

### 2. access-kit 无障碍支持框架

**两级过滤器：**
- 第一级 ✅✅：AI 做不好（无障碍需要 empathy + 真实用户测试），方法论/标准 3 年后仍有价值
- 第二级 ✅✅⚠️：强判断型资产（设计决策、审计方法论）、但"社区活着吗"取决于你从零建还是加入已有项目

**瓶颈分析——"测试"是问题还是护城河？**
你提到"瓶颈在于测试"。换个角度：**测试恰好是护城河**。

- AI 可以生成 ARIA label、焦点管理代码 → 代码层贬值
- AI 不能坐在屏幕前闭眼用屏幕阅读器测试 → 测试层天然抗 AI
- 没有真实用户反馈 → 无障碍实现的质量无法验证

所以"测试是瓶颈"不是 bug，是 feature。问题是：**你有测试条件吗？**

**个人匹配度的核心矛盾：**
- 你有 a11y 积累（`drafts/a11y-open-problems.md`）
- 但你的项目诊断说"取决于你是否玩那游戏"→ 这里你说的是"通用框架"，规模大了很多
- 与 ChemE/PSE 主线零交集
- **需要外部用户反馈**不符合你框架中 P1 的条件（"不需要复杂外部反馈"）

**判定：** 战略方向正确，但对你个人而言有三个问题：(1) 与 ChemE 主线无关，(2) 通用框架维护负担大，(3) 缺乏验证闭环（谁能帮你测？）。如果要做，按项目诊断的建议：**选一个你实际使用的具体场景，做一个具体改进，不建通用框架**。

### 3. DLMF 函数完整实现（Rust/Julia）

**两级过滤器：**
- 第一级 ✅✅：AI 能生成单个函数实现，但**不能替代算法选择、精度权衡、审美判断**——这些是核心产出。测试集/基准 3 年后仍有价值。
- 第二级 ✅✅✅：判断型资产（选算法=判断，测试集=规格），Julia 社区活跃，你有独特领域知识

**这就是"特殊函数手表匠"（项目诊断候选 1）：**

每个函数是自包含的"机芯"：
```
选函数 → 读原始论文 → 选算法 → 实现 → 跑误差测试 → 画误差曲面 → 写笔记
```

**为什么这个最适合"缓解 AI 焦虑"：**

1. **AI 拿不走核心乐趣**：AI 可以生成 `erf` 实现，但不能替你欣赏 1969 年 Cody 的 table-based 方法和 1993 年 Tang 的 polynomial approximation 哪种更优雅。品味和判断是手艺的核心。

2. **进度立即可见**：一个函数跑通 = 一个完成。不依赖外部反馈。

3. **积累不可替代**：你对 100+ 个函数的算法选择、精度权衡、边界 case 的理解，形成一个 AI 训练数据中没有的知识体系。

4. **与已有身份一致**：你维护过 C/F77 数学库，有 Julia triage 身份。这是"你已经在做的事"的自然延伸。

5. **AI 是工具不是威胁**：你可以用 AI 生成样板代码、写测试、生成文档——但算法选择的判断是你的。这是**用 AI 放大**的完美场景。

**潜在问题：**
- 偏内向——几乎没有外部反馈。但作为"工艺放松项目"这不是致命问题（木匠也不需要掌声）
- 与 ChemE 主线的关联较弱（但"物性手表匠"可以把这桥接起来——见下文）

**判定：** ⭐⭐⭐⭐⭐ 最适合作为长期闲暇投入的手艺项目。

### 4. core-math f64 随机测试

**两级过滤器：**
- 第一级 ✅✅：AI 可以生成测试，但**定义"什么行为算正确"和发现新边界 case** 是判断，AI 做不好
- 第二级 ✅✅⚠️：判断型资产（测试集、bug 报告），社区有但你需要选目标

**这本质上是"浮点陷阱图鉴"（项目诊断候选 2）+ "Julia 测试补全"（候选 5）的混合：**

- 对特定版本的 core-math 做随机测试 → 发现边界 case → 最小化复现 → 归档
- 每个发现的 bug 是一个独立产出
- 天然有"田野调查"的手艺感

**优势：**
- 启动成本极低——下载 core-math，写个 fuzzer，开始跑
- 每个 bug 发现 = 明确的正反馈
- 可以直接贡献回 Julia/核心数学库社区
- 和 DLMF 函数实现互补：实现函数时积累的理解帮助设计更好的测试

**缺陷：**
- 发现 bug 是不确定的——可能跑一周什么都没发现（类似证明搜索的阻塞问题，但程度轻得多）
- 不如实现函数有"造物"的审美回报

**判定：** ⭐⭐⭐⭐ 优秀。最适合作为 DLMF 函数实现的**并行辅助线**——做实现时积累的理解用于设计测试，跑测试时发现的 bug 反哺实现。

### 5. libm 实现发展史博客/视频

**两级过滤器：**
- 第一级 ✅✅：策展/历史分析是判断，AI 可生成单篇但不能做"考古"——理解为什么 1979 年的作者用了那个奇怪的常数
- 第二级 ✅✅✅：判断型资产，个人有独特领域知识

**关键定位问题：**

这个项目不应该独立存在。它天然是**其他项目的"输出层"**，不是独立的"生产层"。

正确的关系是：
```
DLMF 函数实现（手艺线） → 积累理解 → 写成博客/视频（输出线）
浮点陷阱发现（测试线） → 积累案例 → 写成博客/视频（输出线）
```

如果你没有在做实现或测试，单纯从文献中写 libm 历史，会缺乏第一手 experience 带来的洞察。"我实现 `sin` 时发现 Sun 的算法在 denormal 附近有这个问题" vs "根据文献，Sun 的算法在 denormal 附近有这个问题"——前者是不可替代的。

**判定：** ⭐⭐⭐½。作为**配套输出活动**极好，但不适合作为独立主线。每实现一个函数，写一篇"这个函数的故事"——这是最自然的节奏。

### 6. AI驱动挖掘数值计算社区bug

**两级过滤器：**
- 第一级 ✅✅：发现新 bug 是强判断型工作，bug 报告是永久资产
- 第二级 ✅✅✅：强判断型资产，你有独特领域知识

**这是"方法论"不是"独立项目"：**

这个想法本质上是一个**工作方法**，可以应用到项目 1、3、4 中的任何一个：
- 对 core-math 函数用 AI 生成测试 → 发现 bug（项目 4）
- 对 DLMF 实现用 AI 辅助验证 → 发现 bug（项目 3）
- 对形式化规范用 AI 辅助检查 → 发现 gap（项目 1）

"AI 驱动"是你的工具链升级，不是独立的方向。把"AI 驱动"看作**你所有项目的默认工作方式**，而不是一个单独的项目。

**判定：** ⭐⭐⭐⭐。不是一个独立项目，而是一个**元方法论**——建议把它作为所有数值计算相关工作的默认模式，不单独立项。

## 综合推荐：三层结构

基于以上分析，建议一个**三层嵌套结构**：

```
═══════════════════════════════════════════
内核层（手艺主线）：DLMF 函数实现
═══════════════════════════════════════════
  每个函数一个 Sprint（1天-2周）
  产出：实现 + 测试 + 误差分析 + 笔记
  平台：Julia（贡献回 Base.Math / SpecialFunctions.jl）
  品味：算法选择、精度权衡、代码审美
  AI 角色：样板代码、测试生成、文档——你用判断把关

  ╔═══════════════════════════════════════╗
  ║ 辅助线 1：随机测试 / 浮点陷阱挖掘  ║
  ║  与内核共享底层知识                  ║
  ║  产出：bug 报告、边界 case 集         ║
  ╚═══════════════════════════════════════╝

  ╔═══════════════════════════════════════╗
  ║ 辅助线 2：博客/视频输出              ║
  ║  内核每完成一个函数 → 一篇博客       ║
  ║  辅助线每发现一个漂亮陷阱 → 一篇博客 ║
  ╚═══════════════════════════════════════╝

═══════════════════════════════════════════
探索层（Sprint 探路）：形式化验证探路
═══════════════════════════════════════════
  选一个内核层已实现的函数
  尝试用 AI 辅助形式化验证
  产出：一篇文章（成功/失败都有价值）
  判断：验证这条路作为手艺是否可行

═══════════════════════════════════════════
交叉层（ChemE 桥接）：物性计算手表匠
═══════════════════════════════════════════
  和内核层是同一模式的不同领域
  用相同的工作流实现热力学方法
  产出：实现 + 验证 + 笔记
  价值：连接手艺项目与 ChemE 专业身份
```

### 为什么这个结构好

1. **内核层提供稳定的手艺心流**——每个函数是独立的、可完成的、有审美回报的
2. **辅助线提供变化和外部反馈**——发现 bug 是惊喜，博客是社交可见度
3. **探索层消化了"形式化验证"的冲动**——不承诺，先探路
4. **交叉层桥接到专业身份**——让你不会觉得"我在做和 ChemE 完全无关的事"
5. **AI 在每个层面都是工具而非威胁**——你始终在用 AI 放大判断，而非担心 AI 替代你

### 最小启动（本周末可做）

1. 选一个函数——建议从 `erf` 或 `lgamma` 或 `cbrt` 开始
2. 读原始论文/参考实现
3. 手写一个实现
4. 跑误差测试，画误差曲线
5. 写一篇笔记

完成这一轮后，你就知道这个节奏是否适合你。

## 关于 access-kit 的单独判断

access-kit 在你的框架中是一个**特殊案例**：

- 战略价值极高（AI 最不擅长无障碍设计）
- 但个人匹配度有三个红灯：与 ChemE 无关、需要外部用户反馈、通用框架维护负担大

建议：**不建通用框架，而是做具体场景的"无障碍审计 + 实现 + 方法论文章"**。参考项目诊断 E 的结论——选一个你用的东西，做一件事，写一篇报告。如果你将来发现这个模式可以抽象成框架，那时候再考虑 access-kit。不要反过来——先建框架再找场景。

## 一句话总结

> **以 DLMF 函数实现为内核手艺主线，随机测试/浮点陷阱为辅助探索线，博客为输出线；形式化验证做一次 Sprint 探路，不做承诺；access-kit 降级为具体场景的审计+实现，不建通用框架。AI 在每个层面都是加速器，判断力始终是你的护城河。**
