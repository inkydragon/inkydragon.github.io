### 1.2 TEP 实现（已有）

| 实现 | 语言 | 依赖 | 成熟度 |
|------|------|------|--------|
| **原始 FORTRAN** | F77 | 无 | ⭐⭐⭐⭐ |

**关键发现：** 所有现有的 TEP 实现都是"黑箱"——你输入操作变量，得到输出。没有一个提供**可视化的流程结构**、**可交互的单元操作拆解**、或**流程拓扑的可视化展示**。pyTEP 依赖 MATLAB——这对教学和 demo 是巨大障碍。

### 1.3 Web/Browser 方案（已有）

| 项目 | 技术 | 用途 | 成熟度 |
|------|------|------|--------|

| **DWSIM Pro** | Cloud (付费) | 商业流程模拟 | ⭐⭐⭐⭐ |

| **MUSICA** | WASM (Rust) | 大气化学 | ⭐⭐⭐ |

**关键发现：没有免费的、开源的、纯浏览器端的通用化工流程模拟器。** Reactor Lab 最接近但仅限于反应器教学。Modiator 证明了 WASM 技术路线在 Modelica 系统仿真是可行的。

### 1.4 SDG/因果图（已有）

Signed Directed Graph（符号有向图）在化工故障诊断中有 20+ 年的研究历史。核心思想：用有向图表示过程变量间的因果关系（+ 表示正相关，- 表示负相关），通过图传播算法追踪故障的根因。

- Maurya et al. (2004) 在 TEP 上验证了 SDG 方法
- 已有 SDG 自动从流程模型生成的算法
- 和你的"符号化测量点位关联"想法完全对应

### 2.2 核心差异化（四层）

```
═══════════════════════════════════════════════════
可视化层：流程拓扑 2D + 设备 3D + 动态趋势图
═══════════════════════════════════════════════════
符号分析层：SDG 因果图自动生成 + 故障传播分析
═══════════════════════════════════════════════════
LLM 层：自然语言解释 + 流程问答 + 操作建议
═══════════════════════════════════════════════════
仿真核心层：动态物料/能量衡算 + 热力学 + 控制
═══════════════════════════════════════════════════
```

每一层都有独特价值：
- **仿真核心**：这是"基座"，最重、最花时间。和 DWSIM/EMSO 竞争但在浏览器中——独特。
- **符号分析**：SDG 在化工研究中成熟但**从未被集成到通用模拟工具中**。这可以是一个独立模块——给定流程拓扑和操作数据 → 自动生成 SDG → 传播分析 → 输出因果链。
- **LLM 层**：几乎零开发成本（调 API），但体验价值高。
- **可视化层**：纯前端，用 React Flow + Three.js/D3.js。有大量现成库。

## 四、核心风险分析

### 风险 1：规模爆炸（⭐⭐⭐⭐⭐ 最高风险）

"化工动态流程模拟器"是一种"无边界项目"：
- 热力学模型：Peng-Robinson、SRK、NRTL、UNIQUAC、UNIFAC、PC-SAFT……几十种
- 单元操作：泵、压缩机、换热器、反应器、蒸馏塔、吸收塔、萃取塔……几十种
- 数值方法：非线性方程组求解、DAE 积分、优化、参数估计……每种都可以单独做一个 PhD
- 物性数据库：DIPPR 有 2000+ 种纯物质

**DWSIM 经过了 15 年 + 多个贡献者的积累才达到今天的规模。一个人从零开始——需要非常激进的 scope cutting。**

### 风险 2：使用者太少

化工流程模拟器的用户群体极小：
- 化工系学生（用 Aspen/DWSIM 完成作业）
- 工艺工程师（用 Aspen HYSYS/ChemCAD）
- PSE 研究者（用 MATLAB/Python/Julia 自建模型）

即使 DWSIM——最成功的开源流程模拟器——在 15 年的积累后，用户数量仍然远小于一个普通的 Web 框架或游戏引擎。

**你的框架中有一条铁律："自己不用 + 无明确用户 → P3 → 不做"。这里你确实有自用场景（简历 + 研究），但"外部用户"可能趋近于零。**

### 风险 3：工具性劳动量巨大

化工流程模拟器的代码中，大量是"必须写但不创造独特价值"的代码：
- 物性计算（翻手册数据到代码）
- 单元操作的物料/能量衡算（教科书公式）
- 收敛算法（已有成熟方法）
- UI/交互（大量样板）

**这部分 AI 可以加速，但不会消除——你仍然需要验证正确性。而验证正确性恰好是化工模拟中最难的部分（你需要参考数据和已知案例做对比）。**

### 风险 4：TS 不是化工建模的理想语言

用 TypeScript 做化工模拟是一个**原创性选择**，但：
- 现有化工计算生态全部在 Python/FORTRAN/C++——你需要从零实现所有物性计算和热力学
- 没有现成的数值求解器库（DAE 求解器在 TS 生态中几乎不存在）
- WASM 桥接可能是更务实的路径（用 Rust/C 写计算核心，WASM 编译到浏览器，TS 做前端）

**如果你想做"完整的通用模拟器"：TS 是瓶颈。**
**如果你要做"TEP 案例的可视化 demo"：TS 完全够用。**

## 五、最关键的策略选择：定位

这里有一个根本性的分叉：

### 路径 A：通用流程模拟器（重、深、慢）

```
目标：一个可以在浏览器中搭建任意化工流程的工具
范围：热力学 + 单元操作 + 动态仿真 + GUI
对标：DWSIM 的 Web 版
工期：年-数年
风险：极高（规模爆炸 + 用户少 + 维护地狱）
```

### 路径 B：PSE 案例的交互式演示平台（轻、广、快）

```
目标：一个可以展示化工流程动态行为、因果分析、
      和 AI 辅助理解的交互式 Web 平台
范围：3-5 个经典案例（TEP + 精馏 + CSTR + ...）
      每个案例 = 简化模型 + 丰富可视化 + SDG + LLM
工期：月-数月
风险：中等（scope 可控，但需要好的 UX 设计）
```

### 路径 C：模块化的"乐高"——分层推进

```
Phase 1（2-4 周）：TEP 交互式可视化
  - 静态流程拓扑图（React Flow）
  - 动态趋势图（Recharts/D3）
  - 预计算数据（从 pyTEP/Fortran 导出，不实时仿真）

Phase 2（4-8 周）：加入简化仿真
  - 实现 TEP 的简化动态模型（ODE，不在浏览器中做完整 DAE）
  - 可以调整操作变量，看趋势变化
  - 3D 设备模型（Three.js）

Phase 3（4-8 周）：SDG 因果分析模块
  - 从流程拓扑自动生成 SDG
  - 故障传播路径可视化
  - 这是一个独立模块——输入是流程拓扑，输出是因果图

Phase 4（2-4 周）：LLM 解释层
  - 接入 LLM API
  - 用自然语言解释当前流程状态
  - 解释 SDG 因果链的含义

后续：可选扩展新案例
```

**推荐路径 C。** 原因：
1. 每个 Phase 有独立产出（可以 demo、可以写博客、可以写进简历）
2. Phase 1 完成后你就能判断这个方向是否值得继续
3. 每个 Phase 都可以停下来——不会出现"做了一年还没东西可展示"
4. 保留了向路径 A 或路径 B 演进的可能性

## 七、具体建议

### 7.1 如果你选择做——技术栈建议

```
前端框架：React + TypeScript
流程图：React Flow（开源的节点编辑器）
2D 图表：Observable Plot / D3.js / Recharts
3D 渲染：Three.js / React Three Fiber
数值计算核心：Rust → WASM（如果 TS 性能不够）
状态管理：Zustand（轻量）
部署：GitHub Pages / Vercel（纯静态）
```

**关于 TS vs Rust/WASM：**
- TEP 的简化 ODE 模型完全可以在 TS 中求解（用 RK4 或简单的 Euler 方法）
- 如果需要复杂的 DAE 求解器或大规模矩阵运算 → Rust → WASM
- "先从 TS 开始，必要的时候把热力学计算迁移到 WASM"是务实的策略

**关于数值求解器：**
- TypeScript 中可用的 ODE 求解器：[odex](https://github.com/littleredcomputer/odex)（JS 移植的 ODEPACK）、自己写 RK4/DP5 也不难
- DAE 求解器：TypeScript 生态中基本没有。TEP 可以用 ODE 近似（把代数方程替换为快速动态）
- 非线性方程组：简单的 Newton-Raphson 自实现即可

### 7.2 最小可展示版本（Phase 1 的具体定义）

```
✅ 一个静态网页，打开就能看
✅ TEP 的流程拓扑图（React Flow，节点=单元操作，边=物料/能量流）
✅ 每个节点可点击，展示该单元操作的详细参数
✅ 预计算的动态趋势图（从 pyTEP/Fortran 导出数据，JSON 加载到前端）
✅ 一个"操作面板"可以调整 2-3 个关键操作变量
✅ （Phase 1 不做实时仿真——趋势图是预计算的）
```

Phase 1 产出一旦完成，你已经有一个可以展示的简历项目了。

### 7.3 关于 SDG 模块的独立价值

SDG（Signed Directed Graph）因果分析模块可能是整个项目中最有**独立学术价值**的部分：

- 它和仿真核心是**解耦的**——你可以从任何流程模拟器（DWSIM、Aspen、自建）导出流程拓扑，然后输入到 SDG 模块
- SDG 自动生成 + 故障传播路径可视化是一个**独立的研究工具**
- 它也可以被做成一个独立的 npm 包或 Web 组件
- 在化工故障诊断文献中，SDG 方法被广泛研究但**从未被做成可用的开源工具**

这个模块可能是你最好的"学术副产物"——即使流程模拟器本身不完整，SDG 模块可以是完整的、可发表的。

## 八、总结

### 8.1 一句话判断

> 化工流程模拟器是一个**高风险、高回报**的想法。作为"名片项目"它的简历可见度和叙事价值极高，但存在严重的规模爆炸风险。关键不在于做不做，而在于**如何定义"完成"**——建议按分层推进（Phase 1 → 4），每个 Phase 有独立产出和停止点。

### 8.2 和你的框架的对齐

| 判据 | 结果 |
|------|------|
| AI 做不好、短期做不好 | ✅ 通过 |
| 产出 3 年后有价值 | ✅ 通过（案例库、SDG 方法论、验证数据集） |
| 核心产出是判断还是代码 | ⚠️ 初期代码量大，长期判断型（SDG、案例策展） |
| 个人匹配（ChemE + 编程 + AI） | ✅✅✅ **极强**——这是你能做而别人做不了的事 |
| 启动检查表回答清楚 | ⚠️ 需要更明确的 Phase 1 定义 |
| god-food 是否持续 | ✅ 简历/研究/PSE 知识体系化 |

### 8.3 如果做——三不原则

1. **不做"通用"**——只做精选案例（TEP + 2-3 个其他经典流程）。通用 = 永远做不完。
2. **不做"完整热力学"**——每个案例只用该案例需要的热力学模型。不要试图建一个"支持所有物性方法"的引擎。
3. **不做"工业级数值求解"**——用简化的 ODE 模型。工业级 DAE 求解是另一个 PhD 的体量。

### 8.4 如果做——推荐启动顺序

```
Week 1-2：Phase 1 最小版本
  - TEP 流程拓扑图（React Flow，静态）
  - 预计算趋势图
  - 部署到 Vercel
  → 产出：一个可分享的 URL + 一篇博客

Week 3-4：评估
  - 用这个 demo 获取反馈
  - 判断是否继续 Phase 2
  - 如果不继续 → 已有一个可展示的简历项目（Phase 1 本身就有价值）

Week 5-12（如果继续）：Phase 2 简化仿真
  - 实现 TEP 简化 ODE 模型
  - 交互式操作变量调整
  - 3D 设备模型

后续 Phase：SDG → LLM → 新案例
```

### 8.5 和其他项目的关系

这个项目和你之前头脑风暴中的其他项目**不冲突**，因为目标不同：

- **化工模拟器**是"功利线"（简历/研究/可见产出）
- **DLMF 函数实现**是"手艺线"（非功利/放松/心流）
- **误差曲面画廊**是"审美线"（可视化/故事）

三条线共享底层知识（数值计算），但服务于不同的心理需求。化工模拟器是上班，DLMF 是下班——两者可以共存。
