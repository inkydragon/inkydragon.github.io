# 原始的想法


我现在的考虑是：希望有一个可以在 ai 浪潮中保持有一定意义的，可以持续做出贡献的开源项目，作为闲暇时间的投入。用于缓解对于 ai 的压力。

以下是一些设想：
- 某种 ai 驱动的探索项目，比如为 core-math 的函数实现增加形式化验证，或者证明。可以人工辅助或者基于现有的证明论文用 ai 去尝试。
- 一些更实际的、更有用的项目，比如 access-kit 一个通用的无障碍支持框架，目前看似乎瓶颈在于测试？
- 一些个人学习向的项目，比如用 rust 或 julia 实现完整的 DLMF 函数实现。可以是追求完整性覆盖面，也可以是追求最优的性能
- 一些纯粹的随机探索项目，比如针对 core-math f64 函数实现无法穷举定义域的问题，可以尝试对于特定版本进行随机测试，或许能发现潜在的问题
- 输出型的项目，比如探索 libm 实现的发展史，选定特定的函数比如 sin，根据它的不同年代的实现，精度-速度的演进，编写博客，以至于做成视频，ai 作为辅助解读。
- 对于正确性的深入探索，针对数值计算社区，用 ai 驱动+各种测试验证方法，尽可能多的挖掘现有代码的bug

另一个想法，写一个开源的化工动态流程模拟器，第一期以复刻 TEP 过程等经典PSE测试案例为例。
预计使用 TS 编写，作为纯前端无服务的形式，便于展示 demo。
配合部分可视化甚至 3d可视化，可以作为本方向的专业性展示。
扩展部分，可以输出符号化的测量点位关联，比如一个换热器的温度点位和其他的哪些部分相互关联，这个应该是单独的模块，纯符号计算部分。还可以加入 LLM 的扩展用于解释等等。
简单看似乎是一个非常长期的工程，可以作为个人的名片项目
请分析评价这个想法，可以搜索开源的流程模拟工具，有一些做的很好带有本地的GUI界面，还有一些是纯粹的库。像 TEP 这类则是纯粹的针对特殊流程的计算。不是通用的，即使这样会使用的人也很少。

目前手上有一个可供参考的经过工业实践的动态流程模拟框架，使用的是 cpp。或许可以参考他的架构。
短期目标还是把手上的案例变为可以供大家交互的案例。长期来看这个项目就是要编写一个动态流程模拟软件，项目没有终止目标同时也意味着可以一致投入，当然后续会变为案例推进的方式，即有新的案例需求，然后再去实现所需的功能。性能部分除非是速度完全不可以接受，那么还是尽量使用 TS 实现，rust-WASM 作为性能备用。
如果有商业、工业价值那么可以考虑用编译型语言重写。
使用 TS 的主要原因是为了快速迭代，并且部署方便。因为案例的目标清晰，需求明确，所以大部分算法都可以直接重写，即使是神经网络，也可以重写或者使用现成的运行时。
一些额外的想法：
- 或许可以支持特殊的模拟。比如纯管网，水-气体管网模拟
- 更高级别的模拟。例如对于一个化工园区的模拟，甚至是整个石化行业的模拟
- 抽取精髓，提取出有趣的部分作为游戏的框架。目前似乎很少有支持连续工业的模拟。基本都是离散的
- 作为一个学习的框架，重温例如精馏塔等基本单元设备。结合符号化，便于分析一些形式，也算是某种科研的工具
- 项目需要在前期就做好架构设计。合理拆分，便于后期扩展。使用 TS 也是js的工程化比较完善，mono-repo 做的很好，当然前期没必要拆分子包，只需要做好模块划分即可。
- 对于给人操作或查看的gui界面不用过多投入，这个项目的重点是流程模拟本身
- 项目预计采用序贯模块法进行计算，因为现有的资料最多。后续或许考虑使用联立方程法便于进行符号分析
- 可以考虑支持或者接入现有的 API 比如 coco CAPE-OPEN；或者是导出模型到 Modelica 等建模软件。
- 一些子模块的实现，比如无尽的热力学可以作为审美项目
