# 一、目前已经明确的设计方向

## 1. 项目定位

这是一个**静态网站**，用于展示和比较**不同语言、不同库对特殊函数的用户可见 API 及其文档语义**。

它的定位不是：

* 数值精度或正确性 benchmark 站
* 底层实现算法考古站
* 在线计算工具
* 全量文档镜像

它更像一个**接口覆盖与语义差异索引**。

---

## 2. 比较对象

当前已经明确：

**比较单位 = 用户可见的 API 条目及其文档定义的行为。**

也就是说，主要关心的是：

* 某库是否提供该函数
* 对外名称是什么
* 支持哪些数值类型
* 支持哪些定义域/参数范围
* 文档如何描述 branch cut、极点、特殊值、错误处理等行为

默认**不以底层 backend 或内部实现为主比较对象**。
后续如果需要，可以增加“文档语义与实际行为不一致”的记录，但那是扩展层。

---

## 3. 函数粒度

当前倾向已经明确：

* 以 **DLMF / Wolfram 的 canonical function** 为主干
* 特殊函数是主要范围
* 暂不纳入初等函数

对于 `scaled`、`inverse`、`regularized`、特定参数化变体等：

* 它们会被记录在系统中
* 通常作为主干函数下的实现条目或 variant/tag
* 默认**不一定进入主比较视图**
* 因为很多变体缺少统一规范，跨库横向比较价值较弱

换句话说，主比较对象优先是“有相对稳定规范的主干函数”。

---

## 4. 展示结构

当前最稳妥的展示方案是三层：

### A. 主表：实现目录

**一行 = 一个 API 实现条目**

主表负责：

* 搜索
* 筛选
* 排序
* 粗粒度比较
* 进入详情或比较页

主表不适合承载太长的说明文字，应主要放：

* canonical function
* family / chapter
* language
* library
* exported name
* support status
* type support
* domain summary
* variant tags
* links
* evidence / verification status

### B. 比较页：少量库并排比较

**只允许选少量库（建议 2–4 个）**

比较页采用矩阵视图：

* 行：函数或函数族下的可比条目
* 列：选中的库

每个格子只放紧凑状态，例如：

* 支持 / 部分支持 / 不支持 / 未核实
* 类型标签
* 风险标签（branch cut、poles、variant 等）

### C. 详情层：右侧面板或展开区

用于显示不适合直接放在表里的内容，例如：

* 完整 signature
* 参数角色
* 更精确的定义域说明
* branch cut / poles / singularities
* 特殊值与错误处理
* 文档摘录
* 证据来源
* 版本说明

当前倾向是：**尽量用侧栏或展开区，而不是重度 modal**。

---

## 5. 信息密度原则

已经比较清楚的一点是：

这个站点不能依靠“把所有说明都塞进表格”来提高信息密度。
真正的高密度展示应是分层：

### 第一层：扫描层

让用户快速判断：

* 有没有
* 叫什么
* 支持哪些大类类型
* 定义域大类是什么
* 是否有明显语义风险点

### 第二层：判断层

让用户判断“能不能拿来用”：

* 参数顺序
* 具体支持范围
* 部分支持的原因
* 文档是否模糊

### 第三层：证据层

让用户追溯来源：

* 文档链接
* 源码链接
* 原始摘录
* 人工审核/LLM辅助/解析来源

---

## 6. 第一阶段范围限定

当前已经隐含形成的第一阶段边界大致是：

* 全静态站点
* 数据收集与网站展示分离
* 以人工审核后的结构化数据为准
* 主要比较文档行为，不直接比较数值正确性
* 主要关注特殊函数
* 比较页只支持少量库并排
* 不做登录、在线编辑、协作后台
* 不做复杂动态图或知识图谱
* 不做历史版本全量时间轴

---

# 二、当前较合理的数据模型草案

虽然你可能先用 CSV 试收集，但逻辑上目前最适合拆成三层。

## 1. Canonical Function

表示标准函数概念。

建议包含：

* `canonical_id`
* `canonical_name`
* `family`
* `dlmf_chapter`
* `dlmf_anchor`
* `wolfram_ref`
* `comparison_enabled`
* `notes`

其中 `comparison_enabled` 很重要，用来标记：

* 该函数/变体是否默认进入主比较体系

---

## 2. Library

表示语言中的某个库。

建议包含：

* `library_id`
* `language`
* `library_name`
* `homepage`
* `docs_base`
* `repo`
* `version_tested`
* `notes`

---

## 3. API Entry / Implementation Entry

表示真正要比较的 API 条目。

建议包含：

* `entry_id`
* `canonical_id`
* `library_id`
* `exported_name`
* `signature_text`
* `variant_tags`
* `support_status`
* `type_support`
* `domain_summary`
* `domain_notes`
* `behavior_flags`
* `doc_url`
* `source_url`
* `evidence_type`
* `verification_status`
* `last_verified`
* `notes`

这是目前最接近主表展示需求的一层。

---

# 三、目前字段设计的核心原则

你接下来试收集数据时，可以优先验证这几个原则是否成立。

## 1. 展示字段与证据字段分离

主表里只放适合筛选、排序、快速扫读的字段。
原始摘录、长备注、复杂语义不要直接占主表列宽。

## 2. 结构化字段与自由文本分离

例如定义域不要只有长文本，至少应拆成：

* `domain_summary`
* `domain_notes`

否则表格不可筛选。

## 3. 归一化字段与原始字段分离

例如类型支持，不应只保留库自己的写法。
最好同时有：

* 站点统一枚举
* 原始文档表述

## 4. “比较主干”与“记录存在”分离

某些变体应该能录入，但不一定默认参与比较。
这能避免主视图被稀疏、弱规范的条目污染。

---

# 四、目前仍待定的问题

下面这些是你试采数据时最值得重点验证的地方。

## 1. canonical function 的粒度还没完全定

虽然大方向是 DLMF / Wolfram，但仍要落实：

* 一个 canonical function 的最小粒度是什么
* 哪些是主干函数
* 哪些只是 variant/tag
* 哪些变体需要单独立项

这会直接影响 CSV 的主键设计。

---

## 2. variant 的处理规则还没定死

目前只形成了倾向：

* 记录它们
* 但通常不参与主比较

还需要进一步明确：

* 哪类 variant 默认只展示不比较
* 哪类 variant 可以作为可比对象进入比较页
* variant 是独立 canonical function，还是主干函数上的 tag

---

## 3. 定义域表示精度仍待定

你已经说参考 Wolfram，但还没确定具体落法。

这是当前最重要的未决问题之一。
至少要决定：

* 主表里显示到什么粒度
* 是否要离散成有限枚举
* 复杂复分析语义如何存储
* branch cut / poles / analyticity 是否拆独立字段

如果这点不定，数据会很快变成大量不可比较的自由文本。

---

## 4. 类型支持的统一枚举尚未定稿

目前你提到的有：

* f32
* f64
* complex32
* complex64
* BigFloat

但还没完全定：

* 站点内部统一名称是什么
* 是否区分库原生支持与通过泛型/多精度支持
* 是否需要标注“文档提及但未明确列出类型”
* 是否要支持更多类型层次

---

## 5. “部分支持”的判定规则未定

这会严重影响比较质量。

例如“部分支持”可能来自：

* 仅支持实数
* 仅支持部分参数范围
* 仅支持部分变体
* 文档不清楚
* 实现存在明显限制

如果不拆原因码，后续筛选和比较会很模糊。

---

## 6. 参数角色归一还未讨论充分

对于特殊函数，这很重要。
不同库即使都实现了同一函数，参数顺序和参数命名也可能不同。

仍待定：

* 是否要为 canonical function 定义标准参数角色
* API entry 是否需要记录参数映射
* 主表是否显示参数顺序差异，还是只在详情里显示

---

## 7. 行为标签体系尚未定稿

你已经隐含需要记录行为差异，但标签体系还没成型。

例如是否要有：

* branch cut
* poles
* singularities
* special values
* domain error
* exception / NaN / status code
* documented behavior only
* doc/impl mismatch

这一层会影响比较页格子的紧凑表达。

---

## 8. 文档行为与实际行为的关系尚未正式入模

你已经说“更进一步可以比较稳定规范与实际实现行为的不一致性”，但这仍属于扩展方向。

还没定的是：

* 第一阶段是否只记录文档行为
* 是否预留字段存实际观察行为
* 是否预留 mismatch 标记

建议即使第一阶段不用，也最好提前预留位置。

---

## 9. 版本策略尚未完全明确

目前看倾向于只记录一个“当前测试/采集版本”，但还没正式定：

* 是否只保留一个版本
* 是否允许备注“某版本起支持”
* 是否未来要扩展为版本维度比较

这会影响库表与条目表的字段设计。

---

## 10. 主表字段还没有最终裁剪

虽然方向已经清楚，但仍需要通过试采验证：

* 哪些列是高频有用
* 哪些列几乎总为空
* 哪些列内容过长，不适合直接展示
* 哪些列需要拆分成标签/枚举/备注

这正是你现在试收集数据最有价值的地方。

---

# 五、建议你试采数据时重点验证的内容

你现在最适合做的不是“大量采集”，而是拿少量函数、少量库做压力测试。
建议每次试采都观察下面几件事：

## 1. 一行是否能稳定对应一个 API 条目

如果一条记录经常需要拆成两三行，说明粒度不对。

## 2. `domain_summary` 是否能写得既短又有区分度

如果总是只能写成长句，说明定义域字段设计还不够好。

## 3. `variant` 是否会大量污染主比较视图

如果一个主函数下面挂了很多弱规范变体，说明要更早引入“仅记录不比较”的规则。

## 4. 类型支持是否真的能统一

如果不同语言/库写法差异极大，说明需要更强的归一层。

## 5. 文档语义是否足够明确

很多库文档可能并不会明确写出 branch cut、极点、错误处理。
这会影响你是否需要：

* `documented`
* `inferred`
* `unknown`

这样的证据状态。

---

# 六、可以作为下一轮输出目标的东西

等你按设计稿试收了一小批数据后，下一轮最适合整理两份东西：

1. **主表字段清单 v1**
   哪些字段保留、哪些删掉、哪些拆分

2. **录入规则草案 v1**
   canonical function、variant、type_support、domain_summary、support_status 的判定规则

这样后面才能比较稳地进入架构设计，而不是在前端实现时再反复返工。
