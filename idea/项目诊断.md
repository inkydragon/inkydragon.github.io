# 项目诊断：两个项目的投入决策

基于 [[项目选择判据-双视角.md]] 的框架逐层评估。

**评估日期：2026-06-13**

---

## 项目 A：JuliaLang/julia 与官方生态（维护者角色）

### 现状

- 身份：triage 维护者
- 当前投入：偶尔审阅 PR
- 使用方式：日常工具，非主力语言
- 潜在投入想法：开发流程改进（测试、构建加速）、生态工具

---

### 五层判据逐层过

**第一层 · 价值判据**

| 判据 | 结果 |
|------|------|
| 产出属性 | 语言本身：规格+社区共识+测试体系 → **判断型资产** |
| AI 替代期限 | 编程语言生态 ≥ 5年，AI 不可能替代 → **通过** |
| 裁员测试 | 去掉代码，语言规格、社区、package 生态、测试基础设施仍在 → **通过** |

✅ 方向级：通过。

**第二层 · 需求判据**

| 判据 | 结果 |
|------|------|
| AI 时代需求趋势 | 科学计算语言的需求大致**稳定**——AI 降低了语言锁定效应，但科学计算正确性和性能保证的价值没有减弱 |

⚠️ 中性的——未增强也未明显削弱。

**第三层 · 资产判据（Julia 作为项目积累了哪些不可替代资产）**

| 资产 | 不可替代性 |
|------|-----------|
| 语言规格 | ✅ 核心护城河 |
| 测试体系（Base + stdlib + CI） | ✅ 边界 case 的巨大累积 |
| 社区共识/治理制度 | ✅ 人的网络 |
| Package 生态（JuliaHub, General registry） | ⚠️ 部分可被 AI 绕过（语言桥接） |
| 开发工具链（构建系统、测试框架） | ⚠️ 工具型资产，AI 可再生 |

✅ 项目整体资产质量高。

**第四层 · 个人匹配度**

| 判据 | 结果 |
|------|------|
| 我有独特领域知识？ | ❌ Julia 编译器/运行时并非你的领域。你贡献的价值主要在流程和生态工具，而非内核 |
| 能增进深入理解？ | ⚠️ 部分——编译器、数值计算，但与 ChemE PhD 主线**不直接相关** |
| 过程有副产品价值？ | ✅ 构建/测试/CI 知识具备可迁移性 |
| 符合个人优先级哪一层？ | **介于 P1 和 P3 之间** —— 不是 P0（与毕业/研究无关），自己确实在用（P1 条件），但非主力语言（削弱 P1），贡献想法体量较大（容易滑向 P3 "无限维护"） |

⚠️ 个人匹配度中等偏低。核心矛盾：**想法大，但动力锚点弱**。

**第五层 · 退出/降级评估**

| 信号类型 | 检查 |
|----------|------|
| 核心价值可被 AI 替代？ | ❌ 不存在 |
| 社区不活跃？ | ❌ Julia 社区活跃 |
| 自己不再使用？ | ❌ 仍在用（工具角色） |
| 维护成本 > 收益？ | ⚠️ 当前审 PR 的成本可控；大贡献的成本未知 |

---

### 诊断结论：**维持现状 + 有条件限制的短期投入**

**维持：**
- 继续保持 triage 角色，偶尔审阅 PR — 这是一种社区治理贡献，属于"判断型"工作，有价值，成本低
- 继续当作日常工具使用 — 这本身就是对生态的贡献（使用即反馈）

**有条件投入（按优先级）：**

1. **如果你自己会用的生态工具** → 做。标准是你写完之后自己日常会跑它。自用 = god-food，纳入 P1。
2. **可以切成 ≤2 周的 Sprint 的流程改进** → 做。比如"加速 Julia CI 的某个具体瓶颈"有明确完成定义 → P2 短周期作品，做完归档。
3. **大型开发流程改革（测试框架重写、构建系统重构）** → **不做**。除非：① 你转为主力语言，或 ② 它直接服务你的 ChemE 工作流。这类项目体量大、维护负担重、god-food 弱，容易变成"有意义的负债"。

**不做的：**
- 纯粹因为"我有 triage 权限所以应该做"的大项目
- 没有明确用户/自用场景的通用生态工具

## 项目 B：AI 辅助形式化验证数学库 — 重评估（作为手工艺项目）

### 评估视角切换

之前的评估用的是"战略价值"视角——AI 时代竞争力、资产积累、社区贡献。现在用**手工艺（craft）视角**重新评估——这件事作为"编程中的木工活"来说好不好。

**手工艺项目的标准：**

| 维度 | 好 | 差 |
|------|----|----|
| 心流潜力 | 重复但有技巧，能进入心流 | 高风险阻塞（卡一个问题一整天） |
| 工作单元 | 自包含，做完一个是一个 | 互相依赖，必须整体推进 |
| 进度可见 | 做一点就能看到结果 | 长期看不到进展 |
| 深度 | 可以深挖，但不强制 | 浅尝辄止或必须深挖 |
| 审美回报 | 结果本身就是审美对象 | 只有跑通/没跑通两种状态 |
| 维护负担 | 产出静止，不需要持续维护 | 依赖上游，版本漂移 |
| AI 关系 | AI 可辅助也可不用，不影响体验 | AI 替代了核心乐趣 |

---

### 形式化验证数学库，按手工艺标准重评

| 维度 | 评价 | 说明 |
|------|------|------|
| 心流潜力 | ⚠️ 中偏低 | 证明搜索经常卡死——一个 lemma 卡几小时是常态。这不是"重复有技巧"，是"困在迷宫里"。心流需要的是**可预测的挑战**，证明不是 |
| 工作单元 | ✅ 好 | 每个函数自包含 |
| 进度可见 | ❌ 差 | 典型困境：3 小时看起来什么都没产出。要么证明通过了（瞬间），要么卡着（很久） |
| 深度 | ✅ 好 | 无限深 |
| 审美回报 | ⚠️ 中 | 证明通过的瞬间有满足感，但证明本身不是审美对象（除非你是逻辑学家） |
| 维护负担 | ✅ 低 | 证明不需要维护 |
| AI 关系 | ⚠️ 微妙 | 你说得对——AI 削弱了这个项目的核心乐趣。前 AI 时代，手工实现数学库 + 形式化验证是完整的手工艺体验：代码是手艺，证明也是手艺。现在 AI 能生成代码，只剩下证明部分，而证明恰好是**卡住/阻塞**风险最高的部分。AI 辅助证明搜索还在早期，不太靠谱 |

**重评结论：作为手工艺项目，形式化验证数学库有结构性缺陷。**

核心问题不是你做不做得好，而是**证明搜索的随机阻塞特性天然反心流**。它更像"解谜"（puzzle）而非"手艺"（craft）。解谜的乐趣在于偶尔的突破，手艺的乐趣在于持续的打磨——你要的是后者。

> 前 AI 时代它是好选择，因为"写代码"这层手艺活填充了证明搜索之间的等待，提供了持续的满足感。现在这层被 AI 剥离了，剩下的证明部分太硬、太脆，不适合放松。

所以：**在"AI 时代战略项目"框架下它仍然高价值，但在"手工艺放松项目"框架下它不是好选择。** 这两个评价不矛盾，取决于你要什么。

## 寻找手工艺项目：数值计算方向

> 目标：找一个可以长期投入、像木工/钟表匠一样慢慢打磨的编程项目。
> 要求：简单投入→明确产出、偏"体力"（心流型重复+技巧）、可 AI 辅助也可古法手写。

---

### 候选 1：特殊函数手表匠（Numerical Function Gallery）

**做什么：** 每次选一个数学函数（`erf`, `log1p`, `hypot`, `tgamma`, `j0`...），深入研究它、实现它、测试它、记录它。

**为什么像手艺：**

- 每个函数是一个自包含的"机芯"——你可以花一个晚上研究 `lgamma` 的算法选择，下一次研究 `cbrt` 的精度权衡
- 工作流极其"体力"：读原始论文 → 选算法 → 实现 → 跑误差测试 → 画误差曲面图 → 写笔记
- 每一步都有可见产出：一个函数跑通了，误差曲线漂亮了，一篇笔记写完了
- **AI 拿不走手艺乐趣**：AI 可以生成一个 `erf` 实现，但 AI 不能替你欣赏 1969 年 Cody 的 table-based 方法和 1993 年 Tang 的 polynomial approximation 哪种更优雅。**品味和判断是手艺的核心**，AI 恰好碰不到
- 维护负担 ≈ 零（函数实现写完就不需要改）

**与你的匹配度：**

- ✅ 维护过 C/F77 数学库，领域背景扎实
- ✅ Julia 的 `Base.Math` 和 `openspecfun` 等就是现成的参考系和对比对象
- ✅ 可以作为 Julia 社区的长期贡献（改进 Julia 的数学函数实现、测试覆盖率、文档）
- ✅ 天然产出文章系列："一个函数的故事"

**最小启动：** 选一个函数，实现它，写一篇笔记。重复。

**长期性：** C99 有 100+ 个数学函数，加上 IEEE 754-2019 新增的、各个语言扩展的——几百个。够做好几年。

**缺陷：** 可能太内向了——几乎没有外部反馈。作为手工艺项目这不是致命问题（木匠也不需要掌声），但如果你想要社区互动，这个方向比较安静。

---

### 候选 2：浮点陷阱图鉴（FP Trap Encyclopedia）

**做什么：** 系统性地收集、复现、解释、可视化浮点数陷阱——那些让数值程序悄然出错的情况。

**为什么像手艺：**

- 更像"田野调查"——发现一个案例、最小化复现、理解机制、画图解释、归档
- 每个案例自包含，体力感很强：跑实验 → 截结果 → 写说明
- 审美产出很好看：一个精心设计的浮点陷阱可视化本身就是作品
- AI 只能生成"已知的"陷阱，不能真正跑代码发现新案例

**与你的匹配度：**

- ✅ 积累了数学库实现中的边界 case 理解
- ✅ 和"特殊函数手表匠"形成互补——一个做实现，一个做陷阱
- ✅ 教育价值高，可以作为教程/演讲素材
- ⚠️ 比候选 1 更缺乏系统性的"推进感"——更像集邮，不像造表

**长期性：** 无限。浮点陷阱是无限的。

---

### 候选 3：数值方法可视化画廊（Numerical Methods Gallery）

**做什么：** 为经典数值方法制作精美的交互式可视化。牛顿法收敛的动画、病态矩阵对解的影响、Runge-Kutta 稳定性区域、有限差分误差随步长变化的动态图……

**为什么像手艺：**

- 强烈的审美回报——可视化的终点是"好看"，而不是"能跑"
- 每个方法是一页，做完即完工
- 你可以选择"古法"手写 Canvas/SVG 渲染，也可以用 AI 加速
- 有人看——可视化项目天然具有传播性

**与你的匹配度：**

- ✅ 数值方法是你 ChemE 主线的核心工具，可视化增进理解
- ✅ 和简历/教学展示可能产生交集
- ⚠️ 前端工作量，不只是数值计算
- ⚠️ 如果做得太重（全功能 Web 应用），容易变成维护负担

**长期性：** 经典数值方法几十个，每个方法的可视化可以浅做（一天）也可以深做（一周）。

---

### 候选 4：FORTRAN 遗产修复（Legacy FORTRAN Restoration）

**做什么：** 选取 netlib、SLATEC 等经典 FORTRAN 数值库中的子程序，做"考古修复"——理解、翻译到现代语言（Julia）、补测试、写历史注释。

**为什么像手艺：**

- 像修复古籍——每段代码背后有作者的经验、时代假设、未写出的推导
- 翻译 + 注释 + 测试的流水线是纯正的"体力"：读 → 理解 → 写等价实现 → 验证一致性 → 写注释
- AI 可以翻译代码，但 AI 不能做"考古"——理解为什么 1979 年的作者用了这个奇怪的常数，为什么这个 branch 在当时的硬件上是对的
- 审美对象是**翻译的优雅程度**——好的翻译既准确又可读

**与你的匹配度：**

- ✅ 已有的 F77 数学库维护经验直接对接
- ✅ 每个子程序是一个完整的考古单元
- ⚠️ 你最熟悉的数学库恰好是之前决定降级的——注意不要重新激活旧的维护负担

**长期性：** netlib 有几百个子程序，SLATEC 有 1400+。选最有价值的即可。

**关键约束：** 这件事和"继续维护旧 F77 数学库"是不同的两件事。前者是**一次性的翻译+注释产出**，产出后不需要维护；后者是**持续维护原库**。只做前者。

---

### 候选 5：Julia 生态中的数值测试补全

**做什么：** 为 Julia 的 `Base.Math` 或 `SpecialFunctions.jl` 等包系统性地补全测试覆盖率——尤其是边界 case、极端输入、跨平台一致性。

**为什么像手艺：**

- 写一个好测试是设计问题："这个函数的什么行为值得锁定？"
- 体力感：每个函数 → 列出应该测试的属性 → 生成输入 → 写测试 → 跑 → 发现 fail → 调查
- AI 可以生成测试样板，但 AI 不能判断"这个 corner case 是否值得锁死为回归测试"
- 直接利用你的 Julia triage 身份，贡献有可见 impact

**与你的匹配度：**

- ✅ 和 Julia 维护者身份直接衔接
- ✅ 补测试是项目最需要、AI 做得最差的工作
- ✅ 每个 PR 都是一个在社区可见的贡献
- ⚠️ 写测试不如写可视化有审美回报

**长期性：** 只要 Julia 还有数学函数，就永远有测试可写。

## 寻找手工艺项目：不限数学方向

> "如果不限制方向，什么是天然适合做成手工艺的编程项目？"

### 手工艺编程项目的通用模式

好的手工艺编程项目通常满足：

> 核心工作是**对某个确定对象的反复操作**（一个函数、一个优化 pass、一个像素、一个 escape sequence、一个字体……），每次操作的难度可预测，产出立即可见，熟练后进入心流。

这与"解谜型"编程形成对比——解谜是你不知道解法，卡住就是卡住；手艺是你知道方法，把每个细节做好。

---

### 候选 A：Tiny Renderer / Ray Tracer

**做什么：** 从零写一个软件光栅化器或光线追踪器，逐步添加功能。

**手工艺性：⭐⭐⭐⭐**

- 每个渲染特性的添加都带来**立即可见的视觉变化**——加阴影 → 画面变立体，加反射 → 画面变真实
- 经典教程路径清晰（tinyrenderer, Ray Tracing in One Weekend），每一步都能预期结果
- 可 AI 加速也可手写——手写 shader 数学和采样策略本身就是放松
- 审美回报是手工艺项目中最高的之一

**长期性：** 从最简单的三角形到全局光照，阶梯无限。可以随时停下，每个阶段都是完整的。

**与你的匹配度：** ⚠️ 和你已有领域（数值计算、ChemE、Julia）没有直接关联。但这恰好可能是优点——"完全无关的放松项目"不会侵蚀主线的心理预算。

---

### 候选 B：Tiny Compiler / Language

**做什么：** 实现一个小型编程语言。Parser → type checker → codegen → optimizer。

**手工艺性：⭐⭐⭐½**

- 每一步是经典、确定的流水线
- 做优化 pass 特别有"钟表匠"感——加一个 peephole optimization，看生成的代码变短一行
- 深度无限：可以永远加 passes
- ⚠️ 语言设计部分是解谜，实现部分是手艺——前半程可能阻塞

**与你的匹配度：** ✅ 你有 Julia 编译器周边经验（triage），对 LLVM/编译器不陌生。

---

### 候选 C：Regex Engine

**做什么：** 从 NFA/DFA 理论开始，实现一个完整的正则表达式引擎。

**手工艺性：⭐⭐⭐**

- 教科书级清晰：Thompson NFA → DFA → 回溯 → 优化
- 每一步是自包含的
- ⚠️ 范围较小，长期性有限（做完就结束了，不像函数库有几百个入口）

---

### 候选 D：Game Boy / NES Emulator

**做什么：** 实现一个经典游戏机的模拟器。

**手工艺性：⭐⭐⭐⭐**

- 每个指令是一条"木工线"——几百条指令，可以一条一条实现
- 每实现一组指令，就有一个 ROM 能多跑一帧
- 非常体力——CPU 指令集就是一本说明书，照着实现
- 审美回报来自看到游戏画面慢慢成形

**与你的匹配度：** ✅ 你有游戏 mod 背景，对游戏技术栈不陌生。

---

### 候选 E：Code Beautifier / Formatter

**做什么：** 为某个语言实现一个代码格式化工具。

**手工艺性：⭐⭐⭐**

- 格式化规则 = 手艺。每一条规则是一个小设计问题："多长的行应该换行？参数怎么对齐？"
- 有明确的输入→输出，调试直观
- ⚠️ 做完后维护需求不低（语言语法会变）

---

### 候选 F：Terminal Emulator

**做什么：** 实现一个终端模拟器。解析 escape sequence，渲染文本，处理输入。

**手工艺性：⭐⭐⭐⭐**

- Escape sequence 按"条"实现——一条一条做，每条都有明确的规范
- 每实现一批序列，就有一个 TUI 程序多渲染对一部分
- 极度"体力"——VT100/ANSI 规范就是一本操作手册
- ⚠️ 可能滑向"处理无尽边界 case"的地狱

---

### 候选 G：Font Rendering / Typography Engine

**做什么：** 从 TrueType/OpenType 字体文件解析开始，实现字形光栅化、布局、渲染。

**手工艺性：⭐⭐⭐⭐**

- 字体解析 → shaping → 光栅化 → 布局，每一层是经典流水线
- 审美回报极高——看到文字在屏幕上正确渲染本身就是美的
- 和 a11y 有交集（无障碍字体渲染）

---

### 候选 H：Database from Scratch

**做什么：** 实现一个最小关系数据库。B-tree → buffer pool → SQL parser → query planner → executor。

**手工艺性：⭐⭐⭐½**

- B-tree 是实现手艺的经典对象——每个 split/merge 都是一个小挑战
- 每一步自包含
- ⚠️ 整体偏"系统工程"而非"手艺"，很多胶水代码

## 寻找手工艺项目：化工/PSE 方向

> 能不能把手艺活和专业方向拉近？
>
> 可以。ChemE/PSE 的很多经典对象天然就是手工艺的好材料——热力学模型、单元操作、反应网络、过程数据——只是以前没人把它们当"编程手艺"来做。

---

### 候选 I：物性计算手表匠（Property Method Watchmaker）

**做什么：** 逐个实现热力学物性计算方法——状态方程（Peng-Robinson, Soave-Redlich-Kwong, SAFT）、活度系数模型（NRTL, UNIQUAC, UNIFAC）、蒸气压方程（Antoine, Wagner）、粘度/导热系数关联式……

**为什么像手艺：**

- 每个方法是自包含的"公式 → 实现 → 验证"。PR 方程你可以花一个晚上手写，和 NIST 数据对比，画出 P-V 等温线的误差
- 实现一个 EoS 本身就是审美对象——代码里嵌着 1873 年 van der Waals 的物理直觉，Peng-Robinson 1976 年的修正，温度依赖的 α 函数演变史
- 每个方法完成后你可以写一篇"这个方程的故事"——谁提出的、为什么这样设计、数值上有什么坑
- AI 可以生成 PR 方程的代码，但 AI 不能替你欣赏其中的物理直觉，也不能替你感受"手写 van der Waals 循环"的乐趣
- 维护负担 ≈ 零

**与你的匹配度：**

- ✅ 化工热力学是你的领域，每个方程你都学过
- ✅ 和 PSE 主线高度相关——物性计算是任何过程模拟的基石
- ✅ 天然跨到"领域知识结构化"——这些方法的现代化实现和文档本身就是公共品
- ✅ 可以用 Julia 实现，产出测试和文档，直接贡献 Thermodynamics.jl 等 Julia 包的改进
- ⚠️ 纯热力学方向，和软测量/PSE 核心研究稍远

**长期性：** 物性方法几百种。DIPPR 数据库有 2000+ 种纯物质的物性参数。够一辈子。

**最小启动：** 选 Peng-Robinson EoS，手写纯流体实现，画出 CO₂ 的 P-V 等温线，和 NIST 数据对比误差曲线。一晚上可完成。

---

### 候选 J：单元操作微型模拟器（Unit Op Mini Simulator）

**做什么：** 逐个实现经典化工单元操作的极小、精美模拟器。精馏塔（McCabe-Thiele）、CSTR 动力学、管壳式换热器设计、吸收塔……

**为什么像手艺：**

- 每个单元操作是自包含的：给定进料和操作条件，输出设计参数或操作曲线
- McCabe-Thiele 图解法的 GUI 实现本身就是审美对象——你可以手绘平衡线和操作线，拖拽进料位置看理论板数变化
- 可以把"化工设计课上学的东西"变成"可以摸、可以拖、可以看的交互式工具"
- 每个模拟器是一个独立作品，做完即归档，不互相依赖

**与你的匹配度：**

- ✅ 化工本科核心知识，不需要额外学习
- ✅ 和 PSE 有一定距离（偏设计而非操作优化），但属于同一领域
- ⚠️ 如果想要 GUI → 前端工作量
- ⚠️ 如果想要"好看" → 需要一定的可视化技巧

**长期性：** 经典的单元操作 ~20 种。每种可以从简单版本（理想体系）逐渐加入复杂性（非理想、动态、控制）。

**最小启动：** 实现一个 McCabe-Thiele 图解法工具。输入：进料组成、回流比、相对挥发度。输出：理论板数和进料位置。用 ASCII art 在终端画出平衡线和操作线（或者加一个最简单的 plotting）。

---

### 候选 K：过程数据日志诊所（Process Data Log Clinic）

**做什么：** 收集真实或仿真的化工过程操作数据，系统性地"诊断"它们——异常检测、趋势分析、故障签名识别——每次用一个经典或新奇的方法。

**为什么像手艺：**

- 每次"接诊"一条数据记录：拿到数据 → 可视化 → 发现异常 → 诊断原因 → 写报告
- 很像医学影像读片——数据就是"病人"，你的工作是发现病灶
- 每个诊断案例自包含，产出为一篇"病例报告"
- 直接对接你的 PSE 主线（过程监测、软测量、异常诊断）

**与你的匹配度：**

- ✅ 这是你的研究领域——过程监测和异常诊断
- ✅ 可以用 Tennessee Eastman 过程、Pensim 等标准仿真平台生成数据
- ✅ 每个案例可以展示一种诊断方法的局限性
- ✅ 长期积累可以形成"化工过程故障模式库"
- ⚠️ 和研究的边界模糊——需要注意不要和研究工作本身冲突（但作为放松项目，可以挑"非论文级"的简单案例）

**长期性：** 数据集无穷。TE 过程有 20 种故障模式，每种可以在不同操作条件下产生不同症状。

**最小启动：** 下载 TE 过程数据，选一种故障模式，用最简单的 PCA 或控制图方法做诊断，写一篇病例笔记。

---

### 候选 L：化工可视化组件库（ChemE Viz Components）

**做什么：** 为化工/过程工程中常见的数据和模型制作精美的可视化组件。相图的交互式探索、反应路径的动画、精馏塔的组成-温度剖面、过程动态的相图……

**为什么像手艺：**

- 每个可视化是一个自包含的"画作"
- 强烈审美回报——化工数据的可视化可以非常美（相图的拓扑结构、P-T 图的相边界、反应网络的图结构）
- 你可以用 Canvas/SVG 手绘，也可以用 D3/Plotly 等工具
- 和"数值方法可视化画廊"类似，但更聚焦化工领域

**与你的匹配度：**

- ✅ 化工数据可视化是你理解的领域
- ✅ 可以做成 Julia 的 plotting recipe 集合，直接贡献生态
- ⚠️ 前端量可能不小
- ⚠️ 每个可视化的"完成"标准模糊（什么时候算够好看？）

**长期性：** 化工中值得可视化的对象无穷。

---

| 候选 | 心流 | 审美回报 | 长期性 | 与 ChemE 关联 | 与 PSE 关联 | 综合 |
|------|------|---------|--------|-------------|-----------|------|
| I. 物性手表匠 | ✅ | ✅ | 极长 | ⭐ | ⭐ | ⭐⭐⭐⭐ |

### ChemE 方向与通用方向的交叉组合

**物性手表匠 (I)** 和 **特殊函数手表匠 (1)** 是同一模式在不同领域的镜像。两者可以同时推进——化学函数的物性方法和数学函数的数值实现。Julia 是它们共同的平台。

**过程数据诊所 (K)** 和 **浮点陷阱图鉴 (2)** 也是同一模式——"田野调查 + 诊断报告"。前者的数据是化工过程操作日志，后者的数据是浮点运算的边界 case。

> 如果你想要纯手艺放松 → 物性手表匠 (I) 最接近"钟表匠"体验。
>
> 如果你想要手艺 + PSE 研究间的部分关联 → 过程数据诊所 (K)。
>
> 如果你想要审美回报最强 → 化工可视化 (L)。

## 职业规划与简历项目方向

> 上面的手艺项目主打"非功利"——审美、心流、长期投入但不求速效。
> 这一节单独讨论功利层面：哪些项目值得做，因为它们能上简历、反哺研究、服务毕业。

---

### 原则：功利项目和非功利项目不是互斥的

同一个领域可以有两条线并行：
- **手艺线**：慢慢打磨，关 AI，追求审美，不为外部反馈
- **功利线**：聚焦产出，用 AI 加速，追求可展示的结果

两条线共享底层知识。你做物性手表匠时积累的实现能力，可以直接服务于简历项目的快速构建。

---

### 简历项目 · 推荐方向

#### R1：AI 辅助化工过程软测量 Demo

**做什么：** 一个干净的、可复现的软测量建模 pipeline。选取一个标准化工过程（如 TE 过程或脱丁烷塔），用现代 ML 方法（Gaussian Process, XGBoost, 神经网络）构建软测量模型，对比传统 PLS/PCR 方法。

**为什么上简历：**

- 直接展示 PSE 核心能力：过程建模 + 数据驱动方法
- AI 工具链使用（数据预处理、特征工程、模型选择、超参数优化）
- 有明确的可视化产出：预测 vs 实测对比图、误差分布、特征重要性

**关键：** 代码仓库要干净——README 清楚、环境可复现（Docker/conda）、结果可一键复现。面试官不会跑你的代码，但会看你的仓库是否专业。

**AI 角色：** 用 AI 加速样板代码（数据加载、绘图脚本、模型训练框架），把自己的时间留给建模决策和结果解释。

---

#### R2：工业过程异常检测演示系统

**做什么：** 一个小型但完整的异常检测系统。读取过程数据 → 实时监测（PCA/ICA/VAE 等）→ 检测异常 → 诊断原因 → 可视化报警。最好做成一个终端 TUI 或简单的 Web dashboard。

**为什么上简历：**

- 比单纯的模型训练更有"工程感"——展示了你理解系统集成
- 过程监测是 PSE 核心方向，和你的研究直接对齐
- TUI/CLI 工具展示编程能力

**最小版本：** TE 过程的 PCA 监测，终端打印 T² 和 SPE 统计量，超限时变色报警。

---

#### R3：AI Agent 辅助的化工实验设计与分析

**做什么：** 用 AI agent（Claude Code 等）辅助化工过程模拟实验的完整流程：实验设计 → 模拟执行 → 结果分析 → 报告生成。记录整个流程中的 AI 有效/无效用法。

**为什么上简历：**

- 展示 AI 时代的工程能力——不是"会用 AI"，而是"知道 AI 在哪有用、在哪没用"
- 跨领域叙事：ChemE + AI 工具链
- 天然产出文章（"AI Agent 辅助化工过程实验——全流程记录与反思"）

**关键：** 记录要诚实——AI 卡在哪里、哪里需要人类介入、哪里 AI 产生了幻觉——这些反思比"AI 成功做完了"更有价值。

---

#### R4：知识变量构造 Pipeline

**做什么：** 化工过程建模中的"知识变量构造"——从过程机理出发，构造物理上有意义的衍生变量（如传热系数、Damköhler 数、各种无量纲数），用这些变量改进数据驱动模型的预测性能。

**为什么上简历：**

- 展示真正的领域知识——不是调包侠，是理解化工物理的建模者
- 和 PSE 研究直接对齐（软测量、混合建模）
- 可以写成论文或技术报告

---

#### R5：化工领域知识的开源结构化

**做什么：** 将化工工程中的某个子领域（如精馏设计经验规则、换热器选型指南、安全阀选型标准）从散落的教材/手册/标准文档中提取出来，做成可查询的结构化知识库。

**为什么上简历：**

- 展示"AI 时代不可替代"的能力——领域知识的提取和结构化
- 跨领域叙事：ChemE + 知识工程 + AI
- 产出可以是网站、JSON/YAML 知识库、交互式决策树

**最小版本：** 选一个子话题（如"常用换热器类型选择决策树"），提取 10-20 条规则，做成交互式页面或 CLI 工具。

---

### 简历项目 vs 手艺项目 · 对比矩阵

| 维度 | 手艺项目 | 简历项目 |
|------|---------|---------|
| **目标** | 过程即回报（心流、审美、放松） | 结果即回报（可展示、可写进简历） |
| **AI 角色** | 可选工具，"古法"优先 | 加速器，用 AI 放大人类判断 |
| **节奏** | 慢，长期，不赶进度 | 有明确 deadline（求职季、会议投稿） |
| **产出物** | 系列文章/笔记、精美代码 | 干净仓库、demo、文章、论文实验 |
| **审美追求** | 代码本身是审美对象 | 结果（图表、dashboard、结论）是审美对象 |
| **失败价值** | 过程学习即价值 | 需要可展示的结果 |
| **维护** | 零 | 通常也零（demo 做完即归档） |

### 功利线和非功利线的协同

**同一个底层知识，两套产出：**

```
底层：物性计算 / 热力学方法 / 过程数据
  ├── 手艺线（慢、古法、审美）
  │     → 博客系列："一个方程的故事"
  │     → 干净的手写实现（不用 AI）
  │     → Julia 生态贡献（补测试、修 bug）
  │
  └── 功利线（快、AI 加速、产出导向）
        → 简历项目：软测量 demo / 异常检测系统
        → 论文实验：知识变量构造 pipeline
        → 技术文章：AI Agent 辅助化工实验
```

两条线不冲突。手艺线积累的理解力直接服务于功利线的建模决策。功利线的快速探索可以验证手艺线中深入的方向是否值得。

### 关于求职叙事的建议

**你已有的叙事线：**

1. **ChemE PhD / PSE** — 过程系统工程，过程监测，软测量，数据驱动建模
2. **AI 工具链能力** — Julia 维护者，AI agent 使用经验，自动化实验
3. **开源贡献** — Julia triage，数学库维护，无障碍方向

**可以加强的叙事线：**

4. **AI 时代的化工工程能力** — 不仅仅是"会用 AI"，而是"理解 AI 在化工建模中的边界"
5. **领域知识结构化的能力** — 把隐式工程知识变成可复现、可查询的公共品

**推荐叙事锚点：** "化工系统工程 + AI 工具链" —— 这个组合在候选人中罕见，且两个方向都在增长。你做的手艺项目（物性手表匠等）虽然不直接写进简历，但它让你在这个叙事中**真的有东西可讲**，而不是编造兴趣。

## 项目 C：RISC-V 全链路学习（开源课程）

### 现状

- 阶段：构想
- 方向：从硬件设计到上层软件的系统性 RISC-V 学习课程
- 性质：教育内容策展 + 个人学习项目

---

### 五层判据

**第一层 · 价值判据**

| 判据 | 结果 |
|------|------|
| 产出属性 | 课程结构 + 知识路径策展 + 练习设计 → **半判断型**（教育设计是判断，教程内容 AI 可生成） |
| AI 替代期限 | AI 能生成单个教程，但策展连贯的多月学习路径、设计递进练习仍是人类优势，窗口约 2-3 年 |
| 裁员测试 | 去掉具体的代码示例和教程文字，课程大纲、知识点依赖图、练习递进设计仍有价值 → **部分通过** |

⚠️ 方向级：边缘通过。教育策展有价值，但 AI 快速侵蚀单篇教程的生产。

**第二层 · 需求判据**

| 判据 | 结果 |
|------|------|
| AI 时代需求趋势 | RISC-V 生态在增长（开源硬件潮流），但 AI 让"从零学 RISC-V"的路径变多——需求**稳定**，非增强 |

**第三层 · 资产判据（未来可能积累）**

| 潜在资产 | 可替代性 |
|----------|---------|
| 知识依赖图（先学A才能懂B） | ⚠️ AI 可生成合理的路径，但最优路径需要人类学习体验反馈 |
| 递进式练习集 | ✅ 好练习题的设计是判断型工作 |
| 学习笔记/踩坑记录 | ✅ 边界 case，AI 训练数据中不存在 |
| 教程文字 | ❌ AI 可替代 |

**第四层 · 个人匹配度**

| 判据 | 结果 |
|------|------|
| 有独特领域知识？ | ❌ 不是计算机架构专家，是学习者而非教学者 |
| 能增进深入理解？ | ✅ 系统性地学习 RISC-V 对底层理解提升很大 |
| 和 PSE 主线的关系？ | ❌ 零交集 |
| 过程有副产品价值？ | ✅ 学习笔记本身就是交付物 |
| 符合个人优先级哪一层？ | **P2** — 学习型项目，技术有趣，产出清晰（笔记/文章系列） |

⚠️ 关键约束：你是学习者，不是专家。**这个项目的产出是"学习记录"而非"权威课程"。** 定位不同决定了投入体量和完成标准完全不同。

**第五层 · 风险**

| 风险 | 说明 |
|------|------|
| **范围爆炸** | "从硬件设计到上层软件"——这是计算机本科 3-4 门课的内容。不做切割，永远不会"学完" |
| **与主线无关** | 时间投入对 ChemE PhD 主线无直接回报 |
| **AI 替代教育内容** | 当你写第 3 篇教程时，AI 可能已经能生成更好的版本 |

---

### 诊断结论：**做成学习日志，不成课程。不承诺覆盖全链路。**

**推荐做法：**

> 定位从"开源课程"降级为"个人学习日志"——写给自己和同行学习者看的踩坑笔记，不是写给初学者看的权威教程。

**Sprint 切割：** 选一个极窄起点，比如"从零跑通一个 RISC-V 软核的 FPGA 综合"。一篇笔记涵盖：
- 环境搭建的坑
- 关键概念理解（用自己的话写）
- AI 辅助学习过程中的有效/无效用法

交付物：一篇笔记。做完就停，下一篇来的时候再开。

**不做：**
- 承诺"全链路"覆盖
- 写成教科书/权威课程
- 追求体系完整性

**核心原则：** 学习可以持续，但项目的**交付承诺**不持续——每篇笔记都是一次独立的 Sprint，写一篇归档一篇。

## 项目 D：同人音乐游戏（两个子方向）

### 现状

- D1：音乐可视化视频
- D2：特定音游的可玩谱面
- 性质：创作型 hobby project

---

### D1 · 音乐可视化视频

| 判据 | 结果 |
|------|------|
| 产出属性 | 视频 — 纯审美/创意产出 |
| AI 替代 | AI 视频生成工具（Sora 等）快速演进。但**品味和精确的节奏匹配设计**仍是人类优势 |
| 裁员测试 | ❌ 视频本身即产出，无代码层可剥离 |
| 个人匹配 | 创意爱好，零 ChemE |
| 优先级 | **P2** — "做完即结束" |

**结论：可以做，本身就是短周期。** 选一首歌，做一个视频，发布。完成即归档。不承诺系列化。

---

### D2 · 可玩谱面

| 判据 | 结果 |
|------|------|
| 产出属性 | 谱面数据（choreography）+ playtest 验证 |
| AI 替代 | AI 可以生成"看起来对"的谱面，但**手感、难度曲线、音乐对齐质量**需要人类反复 playtest，AI 无法替代 |
| 裁员测试 | 谱面设计文件 → ✅ 创意设计资产 |
| 个人匹配 | 如果你自己玩那个音游 → god-food 强；不玩 → god-food 零 |
| 优先级 | 自玩 → **P1**（自用工具级）；不玩 → **P3** |

**结论：取决于你是否玩那个游戏。**
- 玩 → 做，因为每次打出自己做的谱子就是 god-food，playtest 本身就是使用
- 不玩 → 不做，纯粹为他人做谱子没有反馈闭环

---

### 两子方向对比

| 维度 | D1 视频 | D2 谱面 |
|------|--------|---------|
| 周期 | 天-周 | 周 |
| 自用 god-food | 弱（做完看一遍） | 强（反复玩） |
| 社区反馈 | 有（视频平台） | 有（音游社区） |
| 前提条件 | 无 | 自己玩那个游戏 |
| AI 抗性 | 中（AI 视频工具演进快） | 高（手感 AI 碰不到） |

## 项目 E：开源游戏无障碍 mod

### 现状

- 方向：为开源游戏添加无障碍支持（屏幕阅读器、色盲模式、输入适配等）
- 你的背景：已有 a11y 相关草稿（`drafts/a11y-open-problems.md`），方向有积累

---

### 五层判据

**第一层 · 价值判据**

| 判据 | 结果 |
|------|------|
| 产出属性 | 无障碍设计决策 + 实现 → **强判断型** — 理解障碍体验、设计解决方案是 AI 最不擅长的工作 |
| AI 替代期限 | ≥ 5年 — 无障碍需要 empathy + 真实用户测试 + 领域标准知识，AI 几乎无法替代设计层 |
| 裁员测试 | 去掉代码，障碍分析方法论、用户需求文档、测试 checklist 仍是高价值资产 → **通过** |

✅ 方向级：强力通过。AI 时代天然高价值。

**第二层 · 需求判据**

| 判据 | 结果 |
|------|------|
| AI 时代需求趋势 | **稳定偏增强** — 开源游戏数量增长，无障碍是系统性问题而非一次性 bug，需求持续存在 |

**第三层 · 资产判据**

| 潜在资产 | 可替代性 |
|----------|---------|
| 无障碍审计方法论 | ✅ 判断型，可复用 |
| 游戏类型的通用适配模式 | ✅ 知识型资产 |
| 具体 mod 代码 | ❌ 可能随游戏版本失效 |

✅ 即使 mod 代码未来失效，方法论和模式知识持续有价值。

**第四层 · 个人匹配度**

| 判据 | 结果 |
|------|------|
| 有独特领域知识？ | ✅ a11y 方向已有积累，理解无障碍标准和痛点 |
| 能增进深入理解？ | ✅ 从理论到实现，深化 a11y 工程能力 |
| 和 PSE 主线关系？ | ❌ 无直接关系。但 a11y 是跨领域通用价值 |
| 过程有副产品价值？ | ✅ a11y 方法论、开源贡献记录 |
| 符合个人优先级哪一层？ | **取决于自用** — 如果你自己玩那个游戏 → P1；如果是为社区做 → P2 但需明确产出 |

**第五层 · 风险**

| 风险 | 说明 |
|------|------|
| **版本漂移** | 游戏更新 → mod 失效 → 维护负担。需要选更新节奏慢的成熟游戏 |
| **受众验证** | 无障碍设计的质量需要真实用户反馈。无反馈 → 缺乏验证闭环 |
| **范围控制** | "让一个游戏无障碍"是庞大工程 — 必须切极小入口 |

---

### 诊断结论：**有条件做。选一个你玩的开源游戏，做一件事，写一篇报告。**

**启动条件（三条满足方可开）：**

1. **选一个你已经在玩的游戏** — 自用即 god-food，版本更新你能第一时间感知
2. **只做一项无障碍改进** — 比如只加屏幕阅读器支持，或只做色盲模式，或只做键盘完全可操作
3. **产出定义：一篇文章 + 一个 PR** — 文章描述障碍分析方法、解决思路、实现细节；PR 是具体代码

**推荐切入点：** 选一个界面简单、更新慢的开源游戏（如经典 roguelike、棋牌类），先做无障碍审计笔记（只分析，不改代码），再决定是否实现。

**不做：**
- 承诺"完整无障碍化"
- 自己不会玩的游戏
- 频繁更新的游戏（维护成本 >> 贡献价值）

## 项目 F：Godot PuerTS 环境接入（JS 支持）

### 现状

- 自己已定性为：**技术审美项目**
- 方向：为 Godot 引擎添加 PuerTS（JS/TS 脚本）支持
- 性质：引擎集成/胶水代码

---

### 五层判据

**第一层 · 价值判据**

| 判据 | 结果 |
|------|------|
| 产出属性 | 代码（胶水层 + 绑定 + 构建系统）→ **代码型资产** |
| AI 替代期限 | ≤ 1年 — 集成胶水代码是 AI 的舒适区 |
| 裁员测试 | 去掉代码剩下什么？API 映射设计经验 → 很薄。Godot/PuerTS 更新后映射可能完全不同 → **不通过** |

❌ 方向级：不通过。纯代码型，AI 高效替代。

**第二层 · 需求判据**

| 判据 | 结果 |
|------|------|
| AI 时代需求趋势 | **中性偏弱** — JS 在 Godot 中的需求一直存在但不主流（GDScript/C# 占主导），AI 让"学 GDScript"成本降低，JS 支持的需求反而减弱 |

**第三层 · 资产判据**

| 潜在资产 | 可替代性 |
|----------|---------|
| API 绑定代码 | ❌ 随版本失效 |
| 构建/集成脚本 | ❌ AI 可重新生成 |
| 文档 | ❌ AI 可生成 |

❌ 无长期积累型资产。

**第四层 · 个人匹配度**

| 判据 | 结果 |
|------|------|
| 有独特领域知识？ | ⚠️ 有 JS mod 经验（Unity mod 环境），但这和 PuerTS-Godot 集成是不同事 |
| 能增进深入理解？ | ⚠️ 引擎插件机制、语言绑定 — 学习价值中等 |
| 自用吗？ | **关键问题** — 你用 Godot 吗？即使用了，主力语言是 JS 吗？ |
| 符合个人优先级哪一层？ | 自用 + 主力 JS → **P1**；不用 Godot 或不主力 JS → **P3** |

**第五层 · 风险**

| 风险 | 说明 |
|------|------|
| **维护地狱** | Godot 版本迭代 + PuerTS 版本迭代 = 双倍维护。每次上游更新你的绑定可能全部失效 |
| **无 god-food** | 如果自己不用 Godot 或不用 JS 写 Godot 游戏，这个项目唯一的回报是"做出来了" |
| **技术审美陷阱** | 最典型的"有趣但会变成负债"的项目 |

---

### 诊断结论：**强烈倾向不做。除非你和 Godot + JS 存在真实的日常关系。**

**你自己已经给出了正确判断：技术审美项目。**

按照 [[个人项目选择规则.md]] 第四条，技术审美项目允许，但必须满足"做完即结束"。问题是：**引擎集成项目天然做不到"做完即结束"**——上游更新意味着你永远做不完。

**唯一值得做的场景：**
- 你已经在用 Godot 做游戏，且用 JS 作为主力脚本语言 → P1 自用工具
- 你只写了一个最小 PoC（一个 demo 场景 + 一篇博客） → 可以，作为 P2 技术作品

**否则：不进。** 这和你的 Unity JS mod 不同——那个项目你做完了、技术上闭环了、没有上游依赖。PuerTS-Godot 集成的上游依赖和持续维护负担完全不同。
